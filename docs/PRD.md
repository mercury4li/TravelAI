## 1. 背景与机会

### 1.1 问题

旅行者在「种草 → 结构化行程 → 可执行 → 可协作」链路上，常见痛点包括：

- **纯对话类 AI**（如豆包）：出稿快，但**事实与时刻**易错，难以形成可版本化的行程资产，也缺少与地图/同行者的结构化协作。  
- **攻略导入类**（如圆周旅迹）：贴合国内内容习惯，但**自动动线质量**与**可验证性**常不足，**预订闭环**弱。  
- **海外 AI 规划**（如 Mindtrip、Travo、Plantrip）：在可视化、导出、部分预订等各有长板，但**国内 POI/内容源/支付习惯**未必匹配，且普遍存在**门票与时刻表**类误差。

### 1.2 机会假设（待验证）

提供一款 **私人 AI 旅行规划助手**：在长期对话中 **个性化沉淀用户偏好**，并以 **可核验的航班 / 车次 / 酒店数据**（对接权威或授权数据源，而非仅靠模型臆测）支撑结构化行程；让用户少踩通用聊天机器人常见的 **事实性错误**。

## 2. 产品愿景与目标

### 2.1 产品愿景（一句话）

TravelPlan 要成为 **更懂你的私人 AI 旅行规划助手**：通过 **长期、多轮的沟通** 理解并记住你的旅行偏好，逐步产出 **更贴合你心意** 的方案；同时在 **航班、列车、酒店** 等关键事实上 **对齐可验证数据源**，显著减少通用对话机器人常见的「瞎编班次与房源」问题。

### 2.2 核心卖点

#### 卖点一：个性化 —「越聊越懂你」

- **私人助手心智**：区别于「一问一答、单次生成攻略」，产品鼓励用户与助手 **持续交流**——目的地变化、同行人、预算松紧、节奏快慢都会在多轮对话中被逐步厘清。  
- **偏好因人而异**：针对不同用户沉淀 **可结构化 / 可更新的偏好画像**（如交通偏好、住宿档次、餐饮禁忌、景点类型、体力与作息、亲子/老人模式等，**具体维度与存储策略见技术方案**），使后续每一版规划 **越来越贴近该用户的心仪选项**。  
- **渐进式打磨**：支持从模糊想法到可执行草案的 **多轮修订**；用户的纠正、勾选与否定均应 **反哺** 后续推荐与生成逻辑，降低模板感与重复劳动。

#### 卖点二：准确性 —「交通住宿不靠编」

- **对标痛点**：通用大模型（如豆包）在旅行场景中易出现 **推荐使用不存在的车次或航班、枢纽车站混淆、酒店信息与真实库存脱节** 等 **事实性错误**，损害信任。  
- **产品原则**：涉及 **飞机航班、列车车次、酒店** 的展示或推荐时，TravelPlan **不以纯模型幻觉充当事实**；应以 **授权数据接口、官方/聚合查询结果或可核验引用** 为唯一可信来源，并在界面层展示 **数据来源、查询时间与跳转核验路径**（具体接入清单与优先级 **待定**）。  
- **无数据时的诚实**：若某一时段无法查到可靠结果，须 **明确标注「未核验」或「暂无实时数据」**，并引导用户至官方渠道核实，**禁止**伪装成已验证事实。  
- **与闭环预订的关系**：准确性目标的达成 **不依赖** 必须在站内完成支付；MVP 可采用 **查询展示 + 外链深链预订**。若阶段内尚无法接入权威交通/住宿数据，则对外传播须 **弱化「准确性」表述**，避免与卖点二冲突。

## 3. 终端与渠道策略（已确认）


| 阶段               | 终端 / 渠道      | 说明                                                                |
| ---------------- | ------------ | ----------------------------------------------------------------- |
| **MVP**          | **Web（浏览器）** | 桌面与移动浏览器均可访问；优先保证 **响应式 / 移动 Web** 可用，以便行前在手机浏览器完成「看地图、改行程、分享链接」。 |
| **迭代（Post-MVP）** | **飞书小程序**    | 面向办公场景渗透与内部分享；账号体系可与飞书身份对接 **[技术方案待定]**。                          |
| **迭代（Post-MVP）** | **微信小程序**    | 面向微信内裂变、分享与支付习惯（若后续商业化）；需遵循各平台审核与地图 SDK 规范 **[技术方案待定]**。          |


**原则**：业务与数据 API **与前端形态解耦**，小程序阶段尽量 **复用同一套后端与行程模型**，避免为小程序重写核心逻辑。

---

## 4. MVP（Web）

1. **行程 + 地图**：按「天」管理停留点（名称、可选时段、备注），支持排序；地图展示点位与当日顺序（地图 SDK **待定**）。
2. **对话 + AI**：多轮沟通目的地、天数、偏好 → 生成 / 修改 **结构化** 多天行程（不是纯聊天稿）。
3. **交通与住宿**：展示 **航班、车次、酒店** 的查询结果须带来源或可跳转核验；无数据时标 **未核验**，不虚构班次与报价（接入方式 **待定**）。
4. **分享**：生成 **只读分享链接** 给他人查看。
5. **账号**：登录后 **云端保存** 行程列表（是否允许匿名试用 **待定**）。

## 6. 用户故事与验收要点

本章是 **使用过程中产品应覆盖的典型关切与验收点**（含走查问法），**不是**按 §5 用户画像拆分的维度；同一画像用户在不同对话里也可能只命中其中一部分。下列为高频三类关切，**验收时应用同类问法走查**（措辞可替换，约束须覆盖）。

### 典型关切（用户原话抽象）

1. **拼假与性价比**：「五一想出去玩，最多只能请 3 天假，怎样错峰更省？」
2. **多目的地比较**：「喜欢自然风光，5～7 天不想太远、人别太多；在岘港、云贵、广西里犹豫，各自推荐哪里、为什么？」
3. **客流与小众**：「这些地方客流怎么样？有没有小众、相对不挤的玩法或点位？」

---

### 6.1 Epic：错峰、假期与预算（关切 1）


| ID    | 用户故事                                                                         | 验收要点                                                         |
| ----- | ---------------------------------------------------------------------------- | ------------------------------------------------------------ |
| US-1a | 作为用户，我希望说明 **固定法定假（如五一）+ 额外可请假天数上限**，让助手给出 **错峰出发/返程的日期组合** 与理由，以便少请假、少踩高价峰。 | 回复中明确 **可出行窗口**（含请假逻辑）；若涉及票价/酒店价，须 **可溯源或标「未核验」**；忌编造不存在班次。  |
| US-1b | 作为用户，我希望在同一对话里看到 **经济型取向** 的建议（如前后挪 1～2 天对价差的影响），以便在「少请假」和「更便宜」之间权衡。         | 至少给出 **2 套可对比方案**（例如「请 3 天假」vs「只请 1 天假」）；差异点用条列说明，可接结构化行程草稿。 |


---

### 6.2 Epic：偏好、时长、距离与候选地对比（关切 2）


| ID    | 用户故事                                                                                                            | 验收要点                                                                    |
| ----- | --------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| US-2a | 作为用户，我希望在说明 **偏好（如自然风光）、天数（5～7）、可接受距离、希望人少** 并列出 **多个候选区域** 时，助手按区域给出 **推荐目的地/路线 + 推荐理由**（景观、季节、通达、签证等），以便缩小范围。 | 每个候选区域至少有 **1 条具体推荐** 与 **≥2 条理由**；理由区分事实与主观推断（推断须标注）；可一键写入「多日行程骨架」供编辑。 |
| US-2b | 作为用户，我希望追问「若只选其中一个区域，5～7 天怎么排」时，助手在 **不丢失前文偏好** 的前提下重排行程，以便深度比较。                                                | 多轮对话 **继承** 前文约束；新方案写入结构化行程或清晰覆盖旧草案（行为 **待定** 二选一但须一致）。                 |


---

### 6.3 Epic：客流与小众备选（关切 3）


| ID    | 用户故事                                                   | 验收要点                                                                |
| ----- | ------------------------------------------------------ | ------------------------------------------------------------------- |
| US-3a | 作为用户，我希望在选定区域或景点后，询问 **目标时段大致客流或拥挤风险**，以便决定是否改期或改点。    | 须说明 **判断依据**（历史客流、节假日经验、公开报道等）或标 **「经验推断 / 未核验」**；禁止假装有精确实时人数若无数据源。 |
| US-3b | 作为用户，我希望获得 **同区域内小众或相对不挤** 的替代景点、时段或走法，以便保留体验又降低排队与溢价。 | 每条小众建议附 **为何相对人少**；若与热门点属于替代关系，说明 **取舍**（路程、设施、安全等）。                |


---

### 6.4 Epic：行程骨架、地图与智能草案（基础能力）


| ID    | 用户故事                                                     | 验收要点                                       |
| ----- | -------------------------------------------------------- | ------------------------------------------ |
| US-4a | 作为用户，我希望在确定方向后 **新建多日行程**，并在 **地图** 上看到每日停留点与顺序，以便微调动线。  | 按天管理停留点，支持排序；地图与列表一致；路程时间 **有来源或标待定**。     |
| US-4b | 作为用户，我希望用 **自然语言约束** 一键生成或刷新 **结构化多日草案**（写入行程对象，而非仅聊天窗）。 | 生成结果可落格到 Day/Stop；支持对 **单日** 重生成 **[待定]**。 |

## 8. 关键流程（文字版）

1. **登录** → **新建行程** → 用对话完成 **拼假/错峰、候选地比较、客流与小众** 等决策（见 §6）→ **AI 出草案** 落格 → 在列表/地图上调整停留点。
2. 查看 **交通与住宿** 区块：有来源则信；标「未核验」则自行核实。
3. **复制只读分享链接** → 他人浏览器打开查看。

## 9. 开放问题（需产品评审）

1. ~~**首发形态**~~：**已定为 MVP Web**；飞书 / 微信等渠道见 **§3 终端与渠道**（仅策略表，不做排期承诺）。
2. **MVP 是否包含「攻略导入」**：若包含，先做链接还是文本？
3. **地图与 POI 供应商**：国内优先高德？
4. **商业化**：免费 + 订阅 / 按次 AI / B 端？ **[本版 PRD 不写定价]**
5. **隐私**：行程是否默认私密；分享链接有效期？
